PORTFOLIO IS NOT DECORATION. IT'S DUE DILIGENCE
When a developer maintains a serious portfolio, it's not vanity - it's a high-signal artifact worth attention
A carefully crafted repo is the distilled result of years of architectural decisions, trade-offs, refactors, CI failures, design reviews, performance benchmarks, and debugging. It shows thinking: how they structure systems, write tests, document APIs, and evolve code over time
And it's a gift to the employer: real work, available upfront. A meaningful review rarely takes more than an hour - far less than the time burned across multiple interview rounds
If a hiring team dismisses that without serious consideration, it's a red flag
MY EXPERIENCE
I was working full-time, had a newborn daughter, slept in fragments - still, prepared seriously for interviews
I passed screening, live coding, system design
Then, at the very end, I was asked to implement a take-home assignment
It was not declared upfront.
And it wasn't a "2-hour check" - it required a lock-free data structure, a full test suite, and statistics collection. A serious engineering work
I suggested an alternative:
"I work full-time, I'm parenting, this is exactly why I maintain a portfolio. It contains what you're looking for - docs, benchmarks, tests, CI - just slightly different context"
It was not considered satisfactory
By this moment I suggested not to waste our mutual time to each other
If only this was an isolated case! Before I have spent 16 hours building high-performance servers - only to be ghosted.
BUT AI CHANGED EVERYTHING!
Yes - and portfolios evolved too
If a project was built in the AI era, the signal doesn't disappear. It shifts:
- system design decisions
- decomposition of requirements
- API railguards and trade-offs
- turning a prototype into a system
AI can accelerate implementation. It can't fake engineering judgment over years
And if a repo was built 10+ years ago, pre-AI, the signal is even more direct: the candidate wrote everything from the code to readme - which is exactly the discipline most teams claim they want
ARE THE TEST ASSIGNMENTS STILL REASONABLE?
Sure, for a
- very specific domain constraint
- time-boxed design exercise
- competitive requirement for all candidates
But then:
1. Declare it upfront
2. Define scope and expected hours
3. Be honest about time cost
4. Explain why the portfolio is insufficient
"Because we want to see how you write code" is not an explanation
Then read the code they already wrote!
HIRING IS TWO-WAY EVALUATION
If a company ignores public work but expects unpaid custom implementation, it reveals more about the process than the candidate. Respect for engineering starts with respect for time, clarity, and intellectual work.
If those are missing in the hiring process, they will be missing in the working process too.
